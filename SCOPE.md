# SCOPE.md — Anomaly Log & Database Schema

This document covers two things required by the assignment:

1. **Anomaly log** — every data-quality problem found in `Expenses_Export.csv`,
   how the importer handles it, and why.
2. **Database schema** — the data model used to store the cleaned data.

The anomaly log below is generated by running the importer
(`backend/src/importer/runImport.js`) against the provided CSV. Every row of
this table corresponds to an `AnomalyRecord` written to the database during
import, and the same data is exposed via the **Import Report** in the app
after an upload.

---

  not shared costs)

**Severity legend**

| Severity | Meaning |
|---|---|
| `INFO` | Cosmetic/normalisation fix. No effect on balances. |
| `WARNING` | The importer made a judgement call. The row **is** included in balances, but a human should double-check. |
| `ERROR` | The importer could not confidently process the row. It is **excluded** from balance calculations until fixed. |

## 2. Database schema

See `backend/prisma/schema.prisma` for the full Prisma schema (PostgreSQL).
Summary of the models:

### `Person`
One row per household member (current or past) **and** any one-off guest
encountered during import (e.g. "Dev's friend Kabir"). `isActive` lets people
move out without losing their expense history.

### `Import`
One row per CSV upload. Stores the summary counts that power the **Import
Report** screen (`totalRows`, `expenseCount`, `settlementCount`,
`anomalyCount`, etc).

### `Expense`
A shared cost.
- `amountOriginal` / `currencyOriginal` — exactly as recorded in the source.
- `amountInInr` / `exchangeRateUsed` — normalised value used for all balance
  math (equal to `amountOriginal` when the currency is already INR).
- `splitType` — `EQUAL | PERCENTAGE | SHARE | UNEQUAL`.
- `flagged`, `excludeFromBalances`, `isDuplicateOf` — set by the importer for
  rows needing manual review (duplicates, missing payer, etc).

### `ExpenseSplit`
One row per `(Expense, Person)` pair: how much that person owes for that
expense. `shareValue` keeps the *raw* input (a percentage, a share count, or
an unequal amount) for auditability; `amountOwedInInr` is the resolved figure
used in balance calculations.

### `Settlement`
A direct person-to-person payment (e.g. "Rohan paid Aisha back ₹5000", or
Sam handing Aisha his deposit). These move money between two people and are
**not** split among the group — they directly adjust both people's balances.

### `AnomalyRecord`
One row per anomaly found during an `Import`. This *is* the anomaly log above,
persisted to the database — the same data renders in the Import Report UI and
is the source for this document.

```
Person ──┬─< Expense.paidBy
         ├─< ExpenseSplit.person
         ├─< Settlement.fromPerson
         └─< Settlement.toPerson

Import ──┬─< Expense
         ├─< Settlement
         └─< AnomalyRecord

Expense ──< ExpenseSplit
```
