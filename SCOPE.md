# SCOPE.md — Project Scope, Schema & Anomaly Detection Strategy

This document covers:
1. **Project scope** — functional and non-functional requirements addressed
2. **Database schema** — every table explained at column level
3. **Anomaly detection strategy** — every anomaly type: problem, detection logic, resolution policy, user visibility, and engineering reasoning
4. **Import workflow** — the full pipeline from CSV upload to stored records
5. **Assumptions, known limitations, and future scope**

The anomaly table in §4 is generated by running the importer (`node src/importer/runImport.js ../Expenses_Export.csv`) against the provided `Expenses_Export.csv`. Every row corresponds to an `AnomalyRecord` written to the database during import and rendered in the **Import Report** UI.

---

## 1. Project Scope

### What This Application Does

Flatshare Ledger is a **shared-expense tracking system** designed for a household of 4–7 people. Its primary workflow is:

1. A user uploads a raw CSV expense export (the provided `Expenses_Export.csv` format)
2. The import pipeline normalises every field it can, flags every field it cannot confidently resolve, and classifies each row as either a shared Expense or a direct-payment Settlement
3. An Import Report is produced showing every anomaly by row, field, severity, and action taken
4. The dashboard shows net balances (who owes whom) and a minimal suggested set of settle-up transactions
5. Users can manually add settlements, manage people, and toggle individual expenses in/out of balance calculations

### What This Application Does Not Do

- **No authentication** — single-household deployment, not multi-tenant
- **No live currency rates** — USD→INR conversion uses a fixed documented rate
- **No recurring expense scheduling** — expenses are entered via CSV import or manual form
- **No mobile-native app** — web only (responsive CSS but not a PWA)
- **No bank integration** — no automatic import from payment apps

## 3. Non-Functional Requirements

| Requirement | Approach |
|---|---|
| **Auditability** | Every automated data transformation produces an `AnomalyRecord`. Nothing changes silently. |
| **Reproducibility** | Fixed FX rate, deterministic sorting; the same CSV always produces the same balances |
| **Fail-safe imports** | A bad row is flagged and imported (with exclusion if needed), never causes the entire import to abort |
| **Fail-safe AI** | If `ANTHROPIC_API_KEY` is absent or the Claude API errors, `summarizeAnomalies()` returns `null` and the UI renders without the summary paragraph |
| **Separation of concerns** | Import parsing (`engine.js`) is completely decoupled from DB persistence (`importService.js`) — the CLI runner (`runImport.js`) uses the parser without touching the database |
| **Schema as documentation** | `prisma/schema.prisma` is the single source of truth for the data model; no separate ER diagram to keep in sync |
| **Testability** | Every importer module (names, dates, amounts, splits) is a pure function; the CLI runner provides integration-level testing against the real CSV |

## 4. Database Schema

### 4.1 `people`

Stores every person who has appeared as a payer (`paid_by`) or a split participant (`split_with`) in any imported CSV, plus manually-added people.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | Int | PK, autoincrement | Surrogate key |
| `name` | String | UNIQUE | Canonical name after normalisation (e.g. `"Priya"`, never `"priya"` or `"Priya S"`) |
| `isActive` | Boolean | default `true` | `false` when person has moved out. Expenses and splits are preserved. |
| `joinedAt` | DateTime? | nullable | Move-in date (optional; not populated from CSV import) |
| `leftAt` | DateTime? | nullable | Move-out date. Set from `ROSTER_DEPARTURES` during import if person is a known departure. |
| `notes` | String? | nullable | Free-text notes. Auto-populated for one-off guests: `"Auto-created one-off guest during import."` |
| `createdAt` | DateTime | default `now()` | Record creation timestamp |

**Relations:** `expensesPaid` (Expense[]), `splits` (ExpenseSplit[]), `settlementsSent` (Settlement[]), `settlementsReceived` (Settlement[])

**Design rationale:** Setting `isActive = false` rather than deleting allows historical expense/split records to remain intact and balances to remain correct after a person leaves. The `leftAt` field enables the `INACTIVE_PERSON_IN_SPLIT` anomaly detection that checks whether a post-departure row still includes a former housemate.

---

### 4.2 `imports`

One row per CSV upload. Powers the Import Report screen and provides traceability from any `Expense`, `Settlement`, or `AnomalyRecord` back to the upload event that created it.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | Int | PK, autoincrement | Surrogate key |
| `filename` | String | — | Original filename as uploaded (`req.file.originalname`) |
| `importedAt` | DateTime | default `now()` | Upload timestamp |
| `totalRows` | Int | — | Total data rows in the CSV (excluding header) |
| `expenseCount` | Int | — | Number of `Expense` records created |
| `settlementCount` | Int | — | Number of `Settlement` records created |
| `skippedCount` | Int | — | Reserved for future use (currently always 0; all rows are imported) |
| `anomalyCount` | Int | — | Total number of `AnomalyRecord` rows for this import |

**Relations:** `expenses` (Expense[]), `settlements` (Settlement[]), `anomalies` (AnomalyRecord[])

**Design rationale:** The `Import` model provides the header data for the Import Report and makes it possible to show import history without re-running the pipeline. Re-uploading the same CSV creates a new `Import` row and additional records — imports are additive, not idempotent.

---

### 4.3 `expenses`

The central table. One row per shared household cost.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | Int | PK, autoincrement | Surrogate key |
| `importId` | Int? | FK → imports.id | The import that created this row. `null` for manually-created expenses. |
| `sourceRow` | Int? | nullable | The CSV row number (1-indexed from header) for traceability |
| `date` | DateTime | — | Parsed and normalised expense date |
| `description` | String | — | Expense description (trimmed) |
| `notes` | String? | nullable | Free-text notes from the CSV |
| `paidById` | Int? | FK → people.id | Who paid. `null` when `paid_by` was empty (MISSING_PAYER); row excluded from balances. |
| `amountOriginal` | Decimal | — | Amount exactly as recorded in the source (may be USD, may be `899.995`) |
| `currencyOriginal` | String | default `"INR"` | Currency exactly as recorded |
| `amountInInr` | Decimal | — | Normalised amount used for all balance calculations |
| `exchangeRateUsed` | Decimal? | nullable | FX rate applied (`83` for USD rows). `null` when no conversion was needed. |
| `splitType` | String | — | `EQUAL`, `PERCENTAGE`, `SHARE`, or `UNEQUAL` |
| `isDuplicateOf` | Int? | nullable | If this row is an exact duplicate, the `id` of the original expense |
| `flagged` | Boolean | default `false` | `true` for duplicates, near-duplicates, and rows with WARNING+ anomalies |
| `excludeFromBalances` | Boolean | default `false` | `true` for exact duplicates and MISSING_PAYER rows. Can be toggled manually. |
| `createdAt` | DateTime | default `now()` | Record creation timestamp |

**Relations:** `import` (Import?), `paidBy` (Person?), `splits` (ExpenseSplit[])

**Design rationale:** Keeping `amountOriginal`/`currencyOriginal` alongside `amountInInr` ensures the source data is never lost — a reviewer can always see that `540 USD` became `₹44,820` and verify the rate used. The `excludeFromBalances` flag means a duplicate can be preserved for audit purposes without affecting the \"who owes whom\" answer.

---

### 4.4 `expense_splits`

One row per `(Expense, Person)` pair — the per-person breakdown of how much each participant owes for a given expense.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | Int | PK, autoincrement | Surrogate key |
| `expenseId` | Int | FK → expenses.id (CASCADE DELETE) | The parent expense |
| `personId` | Int | FK → people.id | The participant |
| `shareValue` | Decimal? | nullable | The raw input for auditability: a percentage (e.g. `30`), a share count (e.g. `2`), or an unequal amount (e.g. `700`). `null` for EQUAL splits where no per-person detail was given. |
| `amountOwedInInr` | Decimal | — | The resolved amount this person owes for this expense, in INR, used in balance calculations |

**Unique constraint:** `(expenseId, personId)` — each person appears at most once per expense.

**Design rationale:** Storing `shareValue` alongside `amountOwedInInr` is an explicit audit trail decision. For a PERCENTAGE split, a reviewer can confirm that `shareValue = 27.27` (rescaled from `30%` in a 110%-summing split) correctly maps to `amountOwedInInr = 392.73`. Without `shareValue`, the rescaling would be unverifiable from the stored data.

---

### 4.5 `settlements`

Direct person-to-person payments that are **not** shared among the household. Examples: Rohan repaying Aisha for a loan, Sam paying his move-in deposit to Aisha.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | Int | PK, autoincrement | Surrogate key |
| `importId` | Int? | FK → imports.id | The import that created this row. `null` for manual settlements. |
| `sourceRow` | Int? | nullable | CSV row number for traceability |
| `date` | DateTime | — | Date of the payment |
| `notes` | String? | nullable | Description or notes |
| `fromPersonId` | Int | FK → people.id | Person who made the payment |
| `toPersonId` | Int | FK → people.id | Person who received the payment |
| `amountOriginal` | Decimal | — | Amount as recorded |
| `currencyOriginal` | String | default `"INR"` | Currency as recorded |
| `amountInInr` | Decimal | — | Amount used for balance adjustments |
| `exchangeRateUsed` | Decimal? | nullable | FX rate if conversion was applied |
| `createdAt` | DateTime | default `now()` | Record creation timestamp |

**Design rationale:** Separating Settlement from Expense is a deliberate modelling choice (see DECISIONS.md §6). A 1:1 transfer directly moves money between two people; representing it as a 2-person `Expense` would create a spurious `ExpenseSplit` and complicate the balance formula.

---

### 4.6 `anomaly_records`

Every data-quality issue detected during a CSV import. This table is the authoritative anomaly log — SCOPE.md and the Import Report UI both read from it (or from the same source data).

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | Int | PK, autoincrement | Surrogate key |
| `importId` | Int | FK → imports.id (CASCADE DELETE) | The import that produced this anomaly |
| `sourceRow` | Int | — | CSV row number where the issue was found |
| `field` | String? | nullable | The CSV column with the issue (`"paid_by"`, `"amount"`, `"date"`, etc.) |
| `anomalyType` | String | — | Machine-readable type code (see §5 below) |
| `severity` | String | — | `INFO`, `WARNING`, or `ERROR` |
| `rawValue` | String? | nullable | The original cell value before any transformation |
| `description` | String | — | Human-readable description of the problem found |
| `actionTaken` | String | — | Human-readable description of how the importer resolved it |
| `createdAt` | DateTime | default `now()` | Record creation timestamp |

**Design rationale:** Persisting anomalies in a relational table (rather than, say, a JSON blob on the `Import` row) allows filtering by severity, querying by anomaly type across imports, and building the Import Report UI with standard Prisma queries. The `rawValue` column preserves the evidence — a reviewer can see that `"priya"` became `"Priya"` without having to re-run the importer.

## 5. CSV Import Scope

### Supported Input Format

```
date, description, paid_by, amount, currency, split_type, split_with, split_details, notes
```

- Delimiter: comma
- Header row: required (auto-detected)
- Encoding: UTF-8
- Empty trailing row: ignored

### Tolerances Built Into the Pipeline

| Field | What's Tolerated |
|---|---|
| `date` | DD-MM-YYYY (primary), Mon-DD (no year; year inferred), ambiguous dates (flagged + best-effort) |
| `paid_by` | Case variations, trailing/internal whitespace, known aliases (`Priya S`, `priya`, `rohan `) |
| `amount` | Thousands-separator commas, sub-paisa precision (rounded to 2dp), negatives (credits), zero |
| `currency` | Empty (defaults to INR), `USD` (converted), `INR` (no-op) |
| `split_type` | Case-insensitive; empty (triggers settlement detection heuristic) |
| `split_with` | Semicolon-delimited; each name normalised individually; unknown names auto-created |
| `split_details` | Semicolon-delimited `"Name value"` pairs; ignored when contradicts `split_type` |
| `notes` | Free text; used for settlement keyword detection |

## 6. Anomaly Detection Strategy

Every anomaly type is defined in `backend/src/importer/severity.js`. The table below documents every type that can be generated.

---

### 6.1 NAME_NORMALIZED

| Attribute | Detail |
|---|---|
| **Problem** | Person names in the CSV are inconsistently cased or abbreviated: `"priya"`, `"Priya S"`, `"rohan "` (trailing space) all refer to the same people as `"Priya"` and `"Rohan"`. Without normalisation, these become separate Person records, corrupting balances. |
| **Detection Logic** | Every `paid_by` and each entry in `split_with` is passed through `normalizePersonName()` in `names.js`. The function looks up the trimmed, lowercased input in an explicit alias table (e.g. `"priya s" → "Priya"`). If the canonical name differs from the input after trimming, the flag fires. |
| **Resolution Policy** | The canonical name from the alias table is used. The original value is preserved in `rawValue`. |
| **User Visibility** | `INFO` anomaly in the Import Report. No balance effect. |
| **Reasoning** | An alias table was chosen over fuzzy matching because a household has at most 5–7 people, making false merges a real risk with edit-distance approaches. Every normalisation is logged so the mapping is 100% auditable. |

---

### 6.2 DATE_FORMAT_NORMALIZED

| Attribute | Detail |
|---|---|
| **Problem** | Row 27 has date `"Mar-14"` — a `Mon-DD` format with no year, inconsistent with all other rows which use `DD-MM-YYYY`. |
| **Detection Logic** | `parseDate()` in `dates.js` tries DD-MM-YYYY first. On failure, tries the `Mon-DD` regex `/^([A-Za-z]{3})-(\d{1,2})$/`. If the month abbreviation matches a known month, the format is normalised. |
| **Resolution Policy** | Parsed as `14-03-2026` — the day and month from the value, year 2026 inferred from the surrounding rows (all other dates in the file are 2026). |
| **User Visibility** | `WARNING` — row included in balances, but user should confirm the inferred date is correct. |
| **Reasoning** | Rejecting the row would discard a valid expense (Airport cab, ₹1,100) for a single malformed field. Inferred year 2026 is the only plausible value given the dataset context. |

---

### 6.3 AMBIGUOUS_DATE

| Attribute | Detail |
|---|---|
| **Problem** | Row 34 has date `"04-05-2026"`, which the CSV's own notes flag as ambiguous: "is this April 5 or May 4? format is a mess". Under DD-MM-YYYY it is 4 May 2026; under MM-DD-YYYY it is 5 April 2026. |
| **Detection Logic** | Hard-coded check: if the raw trimmed value is exactly `"04-05-2026"`, an `AMBIGUOUS_DATE` anomaly is emitted regardless of which interpretation is chosen. |
| **Resolution Policy** | Interpreted as DD-MM-YYYY (4 May 2026) for consistency with every other date in the file. The original value is preserved in `rawValue`. |
| **User Visibility** | `WARNING` — included in balances. User should confirm the interpretation. |
| **Reasoning** | A consistent rule (always DD-MM-YYYY) is preferable to a row-by-row judgement call. The flag ensures a human reviews it. Rejecting the row was ruled out — one ambiguous field shouldn't discard a valid expense. |

---

### 6.4 UNPARSEABLE_DATE

| Attribute | Detail |
|---|---|
| **Problem** | A date field value that matches no known format. |
| **Detection Logic** | `parseDate()` returns `new Date(NaN)` if no format matches. The caller checks `Number.isNaN(date.getTime())`. |
| **Resolution Policy** | Row imported with a null-equivalent date, **excluded from balance calculations**, flagged `ERROR`. |
| **User Visibility** | `ERROR` — explicit exclusion from balances. Requires manual fix before the expense affects anyone's balance. |
| **Reasoning** | An expense without a date cannot be placed in the ledger timeline. Exclusion is safer than guessing. |

---

### 6.5 AMOUNT_THOUSANDS_SEPARATOR

| Attribute | Detail |
|---|---|
| **Problem** | Row 7 has amount `"1,200"` — a comma used as a thousands separator, which JavaScript's `Number()` cannot parse. |
| **Detection Logic** | The raw amount string is checked for commas before parsing. If present, all commas are stripped before `Number()` conversion. |
| **Resolution Policy** | Parsed as `1200`. |
| **User Visibility** | `INFO` — cosmetic fix, no balance impact. |
| **Reasoning** | Thousands-separator commas are a universal spreadsheet convention. Rejecting these rows would be overly strict. |

---

### 6.6 AMOUNT_PRECISION

| Attribute | Detail |
|---|---|
| **Problem** | Row 10 has amount `"899.995"` — three decimal places, which has no meaning for a currency. |
| **Detection Logic** | After stripping commas, the number of decimal places in the string representation is counted. If > 2, the anomaly fires. |
| **Resolution Policy** | Rounded to 2 decimal places using `Math.round((value + Number.EPSILON * value + 1e-9) * 100) / 100` — a formulation that avoids the classic `899.995 → 899.99` floating-point rounding error. Result: `900.00`. |
| **User Visibility** | `INFO` — no balance impact. |
| **Reasoning** | Sub-paisa precision in a currency context is almost certainly a floating-point artifact from a formula in the source spreadsheet. Rounding is appropriate and safe. |

---

### 6.7 UNPARSEABLE_AMOUNT

| Attribute | Detail |
|---|---|
| **Problem** | An amount field that cannot be parsed as a number after comma-stripping. |
| **Detection Logic** | `parseAmount()` checks `Number.isNaN(Number(stripped))`. |
| **Resolution Policy** | Amount set to `0`, row **excluded from balance calculations**, flagged `ERROR`. |
| **User Visibility** | `ERROR` — requires manual correction. |
| **Reasoning** | An expense with an unreadable amount cannot affect balances. Zero is a safe placeholder; exclusion flags it for review. |

---

### 6.8 MISSING_CURRENCY

| Attribute | Detail |
|---|---|
| **Problem** | Row 28 has an empty `currency` field (CSV note: "forgot to set currency"). |
| **Detection Logic** | `resolveCurrency()` checks `trimmed === ""`. |
| **Resolution Policy** | Defaulted to `"INR"` — consistent with every other row from the same time period and with the note's implication that the omission was accidental. |
| **User Visibility** | `INFO` — no balance impact. |
| **Reasoning** | INR is the only plausible default given that 38 of 42 rows are INR and the row's own notes indicate accidental omission. |

---

### 6.9 FX_CONVERSION

| Attribute | Detail |
|---|---|
| **Problem** | Several Goa-trip rows (rows 20, 21, 23, 26) are denominated in USD. Balance calculations require a common currency. |
| **Detection Logic** | `convertToInr()` checks if the resolved currency code is `"USD"`. |
| **Resolution Policy** | Converted at fixed rate `1 USD = ₹83`. Original `amountOriginal` and `currencyOriginal` preserved on the `Expense` record. `exchangeRateUsed = 83` stored for full traceability. |
| **User Visibility** | `INFO` — no balance impact beyond the conversion itself. The rate is explicitly documented. |
| **Reasoning** | A live FX API adds an external dependency and produces non-reproducible imports. A fixed documented rate is transparent and stable. See DECISIONS.md §4. |

---

### 6.10 UNKNOWN_CURRENCY

| Attribute | Detail |
|---|---|
| **Problem** | A currency code that is not `"INR"` or `"USD"`. |
| **Detection Logic** | `convertToInr()` falls through to the default case after checking for INR and USD. |
| **Resolution Policy** | Amount treated as INR 1:1 (no conversion applied). Flagged `ERROR` for manual correction. |
| **User Visibility** | `ERROR` — balance may be wrong; requires manual fix. |
| **Reasoning** | Silently treating an unknown currency as INR would produce wrong balances. Flagging loudly while still importing gives the user the opportunity to correct the rate. |

---

### 6.11 NEGATIVE_AMOUNT

| Attribute | Detail |
|---|---|
| **Problem** | Row 26 has amount `-30 USD` — the Parasailing refund. A negative amount is uncommon in an expense ledger and could indicate a data error. |
| **Detection Logic** | After parsing, `amountOriginal < 0` check in `engine.js`. |
| **Resolution Policy** | Imported as a negative-amount expense. Each participant's `amountOwedInInr` in the split is negative (a credit). This correctly offsets the original Parasailing charge in the balance calculation. |
| **User Visibility** | `WARNING` — included in balances. User should confirm the negative amount correctly offsets the original charge. |
| **Reasoning** | The arithmetic works correctly for credits without any special-casing. Special-casing would add code complexity for no mathematical benefit. |

---

### 6.12 ZERO_AMOUNT

| Attribute | Detail |
|---|---|
| **Problem** | Row 31 has amount `0` (CSV note: "counted twice earlier - fixing later"). A zero-amount expense has no effect on balances and is almost certainly a placeholder or data error. |
| **Detection Logic** | `amountOriginal === 0` check after parsing. |
| **Resolution Policy** | Imported with zero effect on balances. Flagged for manual review/deletion. |
| **User Visibility** | `INFO` — no balance impact. |
| **Reasoning** | A zero-amount row is harmless to balances but worth flagging so it can be cleaned up. Rejecting it would lose the original row reference. |

---

### 6.13 MISSING_PAYER

| Attribute | Detail |
|---|---|
| **Problem** | Row 13 has an empty `paid_by` field for "House cleaning supplies" (₹780). Without a payer, the expense cannot increase anyone's balance. |
| **Detection Logic** | `paid_by.trim() === ""` check in `engine.js`. |
| **Resolution Policy** | Imported with `paidById = null`. Row **excluded from balance calculations**. `flagged = true`. |
| **User Visibility** | `ERROR` — the only ERROR in the `Expenses_Export.csv` import. Requires manual assignment of the payer via the Expenses page before this expense will affect anyone's balance. |
| **Reasoning** | Guessing who paid would silently corrupt balances. Exclusion is the only safe approach. |

---

### 6.14 EMPTY_SPLIT_WITH

| Attribute | Detail |
|---|---|
| **Problem** | An expense with no participants in `split_with`. |
| **Detection Logic** | After parsing split_with, `names.length === 0` (not triggered by `Expenses_Export.csv` but defined for robustness). |
| **Resolution Policy** | Row excluded from balances; flagged ERROR. |
| **User Visibility** | `ERROR`. |
| **Reasoning** | An expense with no split participants cannot be divided. |

---

### 6.15 INACTIVE_PERSON_IN_SPLIT

| Attribute | Detail |
|---|---|
| **Problem** | Row 36 (02-04-2026, "Groceries BigBasket") lists Meera in `split_with`, but Meera moved out at the end of March 2026. The row's own note reads: "oops Meera still in the group list". |
| **Detection Logic** | After resolving split participant names, each name is checked against the `ROSTER_DEPARTURES` array in `engine.js`. If the expense date is after a departure's `leftAfter` date and that person is in the split, the anomaly fires. |
| **Resolution Policy** | Meera's split is retained **exactly as recorded** — the data is not altered. Flagged `WARNING`. |
| **User Visibility** | `WARNING` — included in balances with Meera in the split. User must decide whether to re-split 3 ways. |
| **Reasoning** | Auto-removing Meera and redistributing her share would be guessing at intent — perhaps she genuinely used those groceries during her final week. Preserving the data and flagging it is the only safe approach. |

---

### 6.16 NEW_PERSON_AUTO_CREATED

| Attribute | Detail |
|---|---|
| **Problem** | Row 23 lists `"Dev's friend Kabir"` in `split_with`. This name appears nowhere else in the dataset and does not match any known household member alias. |
| **Detection Logic** | When a name in `split_with` is not found in the known-people set AND matches the guest pattern (`/'s |friend|guest/i`), the anomaly fires. |
| **Resolution Policy** | Auto-created as a `Person` record with `isActive = false` and `notes = "Auto-created one-off guest during import."` Kabir's share (₹2,490) affects the balance calculation. |
| **User Visibility** | `INFO` — user should decide whether Kabir should be billed or whether Dev should absorb his share. |
| **Reasoning** | Ignoring the name would either create a mismatch in split totals or silently exclude a participant. Creating an inactive guest record is the least-destructive option. |

---

### 6.17 PERCENTAGE_SUM_MISMATCH

| Attribute | Detail |
|---|---|
| **Problem** | Rows 15 and 32 both have `split_details` percentages summing to 110% (30+30+30+20 = 110), not 100%. |
| **Detection Logic** | In `calculateSplits()` for PERCENTAGE type, the sum of all percentage values is compared to 100. If the absolute difference exceeds 0.01, the anomaly fires. |
| **Resolution Policy** | Each percentage is proportionally rescaled: `pct × (100 / sum)`. For 110%, the factor is `0.9091`. The rescaled values produce correct `amountOwedInInr` values. The original percentages and the rescale factor are included in the anomaly description. |
| **User Visibility** | `WARNING` — included in balances after rescaling. User should verify the intended percentages. |
| **Reasoning** | Silently rescaling without logging was the initial implementation. This was corrected when SCOPE.md drafting revealed the anomaly table had no entries for rows 15 and 32 — a clear signal that a real data-quality issue was being hidden. See AI_USAGE.md for the full account. |

---

### 6.18 UNEQUAL_SUM_MISMATCH

| Attribute | Detail |
|---|---|
| **Problem** | For UNEQUAL splits, the per-person amounts in `split_details` do not sum to the expense total. |
| **Detection Logic** | Sum of all `value` fields from parsed split_details is compared to `amountInInr`. If difference > 0.01, anomaly fires. |
| **Resolution Policy** | `WARNING` logged. The provided unequal amounts are used as-is, which means splits will not sum to the total — a human must correct them. |
| **User Visibility** | `WARNING`. |
| **Reasoning** | Unlike percentages (where proportional rescaling is mathematically clean), unequal amounts represent specific claimed figures. Rescaling them would silently change someone's stated share. |

---

### 6.19 SPLIT_DETAILS_IGNORED

| Attribute | Detail |
|---|---|
| **Problem** | Row 42 ("Furniture for common room") has `split_type = equal` but also `split_details = "Aisha 1; Rohan 1; Priya 1; Sam 1"` — data that is only meaningful for SHARE/PERCENTAGE/UNEQUAL types. The two fields contradict each other. |
| **Detection Logic** | In `calculateSplits()`, if `splitType = EQUAL` but `split_details` is non-empty, the anomaly fires. |
| **Resolution Policy** | `split_type` is authoritative. Amount divided equally among `split_with` participants. `split_details` ignored. Note: for this specific row, 1:1:1:1 shares produce the same result as an equal split — but the rule must be defined independently of this coincidence. |
| **User Visibility** | `WARNING`. |
| **Reasoning** | `split_type` is the structured, intentional field; `split_details` is free-text that may contain stale or incorrect data. Letting free-text override the structured field would create an unpredictable and hard-to-audit precedent. |

---

### 6.20 SETTLEMENT_DETECTED

| Attribute | Detail |
|---|---|
| **Problem** | Two rows describe direct person-to-person payments, not shared costs: row 14 ("Rohan paid Aisha back") and row 38 ("Sam deposit share"). Treating them as expenses would distort splits (a 2-person expense where both people owe money makes no sense for a repayment). |
| **Detection Logic** | A row is classified as a settlement if **either** `split_type` is empty, **or** description+notes match the keyword regex `/deposit\|paid.*back\|reimburs\|settle?ment\|owes?/i` — **and in both cases** `split_with` (minus the payer) names exactly one other person. |
| **Resolution Policy** | Recorded as a `Settlement` record (fromPerson → toPerson, amount). Adjusts exactly two people's balances directly. |
| **User Visibility** | `INFO` — included in balances as a settlement. |
| **Reasoning** | The heuristic is documented and logged as an anomaly every time it fires, making it auditable and correctable if it misclassifies a future row. See AI_USAGE.md for the case where this was initially incomplete. |

---

### 6.21 POSSIBLE_DUPLICATE

| Attribute | Detail |
|---|---|
| **Problem** | Rows 5 and 6 describe the same expense: "Dinner at Marina Bites" and "dinner - marina bites" — same date (08-02-2026), same payer (Dev), same amount (₹3,200). Classic double-entry error. |
| **Detection Logic** | Second pass over all parsed expenses. For each pair (i, j) with the same date: normalize both descriptions (lowercase, strip punctuation, remove filler words, sort remaining words), compare. If normalized descriptions match AND payer/amount/currency also match → exact duplicate. |
| **Resolution Policy** | Both rows imported. The second occurrence (`j`) has `excludeFromBalances = true`, `isDuplicateOf = expense_i.id`, `flagged = true`. The first occurrence is unchanged. |
| **User Visibility** | `WARNING` — balance uses only the first occurrence. User should confirm and delete row 6. |
| **Reasoning** | Auto-deletion would be destructive and irreversible. Exclusion from balances gives the correct answer immediately while preserving the audit trail. |

---

### 6.22 POSSIBLE_DUPLICATE_DIFFERENT_AMOUNTS

| Attribute | Detail |
|---|---|
| **Problem** | Rows 24 and 25 appear to describe the same dinner at Thalassa restaurant on the same date (11-03-2026), but with different payers (Aisha vs Rohan) and different amounts (₹2,400 vs ₹2,450). This could be two people logging the same dinner, or two separate checks. |
| **Detection Logic** | Same date + matching normalized descriptions, but payer or amount differs → near-duplicate. |
| **Resolution Policy** | Both rows imported and **both flagged** (`flagged = true`). Neither is excluded from balances — the importer cannot safely determine which (if either) is correct. |
| **User Visibility** | `WARNING` on both rows. Balances include both. Human must determine the truth. |
| **Reasoning** | The importer cannot know whether ₹2,400 or ₹2,450 is the correct total, or whether both are valid (e.g. two people paid separately at the same restaurant). Guessing which to exclude would corrupt balances. |

## 7. Validation Rules Summary

| Field | Rule | Fallback |
|---|---|---|
| `date` | DD-MM-YYYY primary; Mon-DD (year=2026) secondary | `new Date(NaN)` → ERROR |
| `paid_by` | Alias table lookup → canonical name | `null` → MISSING_PAYER ERROR |
| `amount` | Strip commas; parse as float; round 2dp | `0` → UNPARSEABLE_AMOUNT ERROR |
| `currency` | `INR` or `USD` (case-insensitive) | `INR` if empty → MISSING_CURRENCY INFO |
| `split_type` | One of: equal, percentage, share, unequal (case-insensitive) | Settlement check; else `EQUAL` + UNKNOWN_SPLIT_TYPE ERROR |
| `split_with` | Semicolons; each name through `normalizePersonName()` | Empty → EMPTY_SPLIT_WITH ERROR |
| `split_details` | `Name value; ...`; used only for non-EQUAL types | Ignored + WARNING if split_type=equal |
| Percentage sum | Must equal 100 (tolerance: 0.01) | Rescale + PERCENTAGE_SUM_MISMATCH WARNING |
| Duplicate check | Same date + normalized description + same payer + same amount | Exclude second; flag both |
| Settlement check | Empty split_type OR keyword match, single other person in split_with | Create Settlement record |

    │               ├── amounts.js: parseAmount(row.amount)
    │               ├── amounts.js: resolveCurrency(row.currency)

    │               ├── names.js:   parseSplitWith(row.split_with)
    │               ├── engine.js:  roster departure check

    │

    │
    ├── importService.js: persistImport(prisma, filename, result)

    │       ├── for each expense: prisma.expense.create()

## 9. Assumptions

1. All dates in `Expenses_Export.csv` fall within calendar year **2026**
2. The canonical date format is **DD-MM-YYYY**; all other formats are treated as edge cases
3. **INR** is the default currency for this household's expenses
4. **1 USD = ₹83** is an acceptable fixed conversion rate for the purposes of determining who owes whom (see DECISIONS.md §4)
5. The household person alias table (`names.js`) is complete for the provided dataset; a different dataset would require updating the `ALIASES` object
6. **Meera** departed after 29 March 2026; **Sam** joined in April 2026 — hardcoded in `ROSTER_DEPARTURES` in `engine.js`
7. Each import is **additive** — re-uploading the same CSV creates duplicate records (no idempotency key)
8. `split_type` is more reliable than `split_details` when the two fields contradict each other

## 10. Known Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| Additive imports (no idempotency) | Re-uploading the same CSV doubles all records | Each import is visible in import history; user can inspect and delete duplicate imports |
| Fixed FX rate | Minor balance inaccuracy for USD rows | Rate is a single constant; documented assumption; original amounts preserved |
| ROSTER_DEPARTURES is hardcoded | Won't detect post-departure rows for new movers-out without a code change | Future: store departure dates in the database and query dynamically |
| No authentication | Any user of the deployed app can modify data | Acceptable for household-scale single-tenant deployment |
| Greedy settle-up algorithm | Produces near-optimal (not always minimum) transaction sets | For ≤10 people, greedy produces optimal or near-optimal results in practice |
| No import rollback | If part of a large import fails mid-way, partial data is committed | Each row is individually safe to fail; DB-level transactions mitigate partial failures |
| No CSV schema validation | Assumes the 9-column format; different column order would fail silently | Document the expected schema; validate column names at import start |
